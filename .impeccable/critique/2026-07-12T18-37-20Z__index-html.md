## Anti-Patterns Verdict

**LLM assessment:** Moderate AI-template risk, not a failure. The Shared Kitchen Table identity, real recipe imagery, and task-oriented controls are intentional. The main risk is repetition: warm banners, asymmetrical rounded corners, paper textures, decorative gradients, oversized serif numerals, uppercase kickers, and staggered card animations recur across the app.

**Deterministic scan:** The installed Impeccable detector returned `[]` for `index.html`. No registered markup anti-patterns were found. The detector does not evaluate CSS texture choices, focus management, publishing semantics, or cognitive load.

**Browser evidence:** The evidence pass verified desktop at `1280x720` and mobile at `390x844`: no horizontal overflow, fixed bottom navigation clearance of `96px`, visible Add focus treatment, and no wrapping for the Add action in English or Spanish. The two-column Week grid is now visible without horizontal scrolling, but its variable-height cards can push the editor below the first mobile viewport. No user-visible overlay was claimed because the critique is read-only.

## Overall Impression

The recent changes improved navigation and mobile scanning without disrupting the product’s warm identity. The biggest remaining product risk is trust: a family member can think they saved a draft when they actually published an incomplete recipe to the shared site, and a Spanish user can see fallback content but still fail to find it or understand why safety actions are locked.

## What's Working

- The persistent bilingual Add action is correctly treated as a shell-level action in `index.html:22` and `app.js:938-953`.
- Source-language recipe fallback is deliberate, with safety-dependent actions still disabled when warnings are untranslated in `recipe-library-ui.js:96-151`.
- The two-column mobile Week grid removes horizontal scrolling and keeps all seven days available in one overview at `styles.css:2923-2935`.
- The app has meaningful recovery patterns: local persistence, sync retries, conflicts, local recipe drafts, and undo.

## Priority Issues

### [P1] “Save draft” publishes incomplete recipes to the shared site

The Add flow labels the action “Save draft,” but `submitUploadForm()` calls `saveSharedRecipe()` after validating only the recipe name. Empty ingredients and steps can reach the live family collection at `recipe-form-ui.js:272-301`.

This is a trust and data-quality problem for family members and caretakers. Make local draft saving distinct from publishing: save incomplete work locally first, require name plus ingredients and steps for publish, and use an explicit “Publish to family” action after review.

Suggested commands: `$impeccable harden recipe publishing`, `$impeccable clarify recipe save states`.

### [P1] Spanish fallback content is visible but not searchable

Cards display source-language fallback through `requiredText()` and `displayText()`, but the search haystack still uses only `localizeExact()` at `recipe-library-ui.js:58-67`. A Spanish-dominant user can see an English fallback recipe but fail to find it by the visible name.

Search both locale values and the canonical source text, or add a clear source-language result marker. Keep the current safety behavior.

Suggested command: `$impeccable harden bilingual recipe search`.

### [P1] Safety locks are not self-describing enough

When an allergy warning is untranslated, `addRecipeGroceries` and `markCooked` are disabled at `recipe-library-ui.js:148-151`, but the explanation is rendered in a separate status block and disabled ghost buttons do not receive a distinct visual treatment. The user can see a disabled action without immediately knowing why.

Place the lock explanation adjacent to the affected actions, add explicit disabled styling, and announce the reason through an accessible status relationship. Do not weaken the safety lock.

Suggested commands: `$impeccable harden safety states`, `$impeccable clarify translation warnings`.

### [P1] The mobile header has a conditional overflow risk

The Add action fits at the tested `390px` width, but `.header-actions` is a non-wrapping flex row at `styles.css:148-153`. When the install prompt is visible, it adds install and dismiss controls to Quick Guide, Add, and EN/ES. Verify `320px` and `375px` with the prompt visible; use a deliberate second row or move installation guidance into Quick Guide if needed.

Suggested command: `$impeccable adapt mobile header`.

### [P2] Week selection and editing remain separated on mobile

Selecting a day rerenders the grid and scrolls to a separate editor at `schedule-ui.js:121-132`. The new grid is better than the old horizontal strip, but long Spanish meal names can create uneven card heights and push the editor far below the initial viewport.

Consider consistent-height summaries with line clamping, an inline selected-day expansion, or a sticky selected-day heading that remains visible above the editor.

Suggested command: `$impeccable adapt Week planning`.

## Persona Red Flags

- **Family member:** “Save draft” sounds private but can publish incomplete content to everyone; the default Family attribution can also obscure who made a change.
- **Caretaker:** Adding inventory still requires At Home, Manage, Add item; grocery utilities remain inside List tools. These are manageable but slow during handoffs.
- **Spanish-dominant user:** Fallback content is safer than a blank recipe, but search excludes fallback text and untranslated warning locks are not explained next to the disabled actions.

## Minor Observations

- Repeated uppercase view kickers, per-view color banners, and the body stripe texture make the app feel more like a generated design system than a calm household tool.
- The body and empty Today hero use repeating gradients, which conflict with the product register’s anti-pattern guidance.
- The calendar disclosure uses the serif display face for a utility control at `styles.css:1107-1117`; the UI face would better match the product guidance.
- Schedule and calendar selection use `aria-pressed`, but there is no live announcement that the editor changed to a new selected day.
- Translation failures are stored in a session-only failed set, so “Try again shortly” may not retry without a reload.

## Questions to Consider

- Is “Save draft” truly intended to publish immediately, or should publishing be a separate reviewed step?
- Should fallback recipes explicitly say “English source” instead of relying on a generic translation status?
- What is the smallest useful mobile handoff: seven visible day cards, or one selected day with clear next/previous controls?
- If caretakers need Quick Guide during a handoff, why does it disappear when they navigate away from Today?
